# Repository Guidelines

## Project Structure & Module Organization

This repository is a TypeScript MCP server for delegating Codex work to Claude Code.

- `src/` contains the server and implementation modules. `server.ts` registers MCP tools, `claude-cli.ts` wraps Claude CLI/job operations, and `schema.ts` defines Zod schemas.
- `tests/` contains Vitest unit tests named `*.test.ts`, generally matching the source module under test.
- `plugins/codex-claude-delegate/` contains packaged plugin metadata, skills, MCP config, and hooks.
- `dist/` is generated by `npm run build`; do not edit it directly.
- `.codex-claude-delegate/`, `.claude/`, and `.debug-fixtures/` are local/runtime or fixture directories.

## Build, Test, and Development Commands

- `npm install` installs dependencies.
- `npm run dev` starts `src/server.ts` using `tsx`.
- `npm run build` compiles TypeScript into `dist/`.
- `npm start` runs the built server at `dist/server.js`.
- `npm test` runs the full Vitest suite once.
- `npm run test:watch` runs Vitest in watch mode.
- `npm run typecheck` verifies types without emitting files.

Run `npm run build` and `npm test` before publishing or changing tool behavior.

## Coding Style & Naming Conventions

Use ESM TypeScript with strict type checking. Keep imports explicit and include `.js` extensions for local runtime imports, matching the Node16 module configuration. Prefer small modules and typed return objects.

Use two-space indentation, double quotes, semicolons, and `camelCase` for variables/functions. Use `PascalCase` for exported types/classes and `UPPER_SNAKE_CASE` for constants. Keep MCP tool names and wire fields in `snake_case`.

## Testing Guidelines

Tests use Vitest with the Node environment and live in `tests/**/*.test.ts`. Name tests after behavior. Mock CLI, filesystem, and process boundaries where possible; avoid real Claude dependencies unless explicitly isolated.

For targeted work, run `npx vitest run tests/<name>.test.ts`. For final verification, run `npm test` and `npm run typecheck`.

## Commit & Pull Request Guidelines

Recent history uses concise messages, often Conventional Commit style such as `fix: throttle background job polling` and `docs: rewrite README for user-facing conciseness`. Prefer `type: imperative summary` (`fix:`, `docs:`, `chore:`, `test:`).

Pull requests should include the behavior change, commands run, and any MCP/plugin compatibility notes. Link issues when available. Include screenshots only for documentation or plugin UI changes.

## Security & Configuration Tips

Do not commit credentials, local Codex config, or Claude session data. Keep allow-root behavior, environment sanitization, recursion guards, and worktree safety checks covered by tests.
