# REL-003 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a repeatable post-publish metadata check that verifies local package metadata, npm registry `latest`, and Git tag alignment for manual release workflows.

**Architecture:** Add a standalone Node.js script that reads `package.json`, queries npm metadata and Git tags through injectable commands, and fails closed on mismatch. Wire it into `package.json` as a maintainer command and document it in the release checklist after publish/tag steps.

**Tech Stack:** Node.js ESM script, `execFileSync`, Vitest subprocess tests, npm scripts.

---

## 1. Background And Goal

Current behavior:

- `check:release` verifies tarball contents and local package/plugin metadata.
- `check:release:install` verifies an installed packed tarball and CLI version output.
- After `npm publish`, registry and Git tag checks are documented as manual shell commands.
- Manual release can drift if the package publishes but the tag is missing, points at a different commit, or registry `latest` does not match local `package.json`.

Target behavior:

- Add `npm run check:release:metadata` for post-publish verification.
- The check must compare:
  - `package.json` name and version.
  - `npm view <package> version dist-tags.latest --json`.
  - `git tag --points-at HEAD v<version>`.
- The script must fail with clear messages for version mismatch, latest mismatch, missing tag, missing package metadata, and invalid command output.

Out of scope:

- Do not add automatic `npm publish`.
- Do not mutate Git tags.
- Do not make this part of `prepublishOnly`, because registry/tag alignment is a post-publish state.
- Do not hand-edit `dist/` or generated plugin server bundles.

## 2. File Impact Scope

`scripts/check-release-metadata.mjs`
- New post-publish metadata verification script.
- Reads local `package.json` from `CHECK_RELEASE_REPO_ROOT` or repo root.
- Accepts test overrides:
  - `CHECK_RELEASE_NPM_VIEW_JSON` for npm output fixture.
  - `CHECK_RELEASE_GIT_TAGS` for tag output fixture.
  - `CHECK_RELEASE_SKIP_NPM=1` to skip registry check for local dry runs.
  - `CHECK_RELEASE_SKIP_GIT=1` to skip tag check for local dry runs.

`tests/check-release-metadata.test.ts`
- New subprocess tests for success and failure modes.

`package.json`
- Add `check:release:metadata`.

`README.md`
- Update maintainer release checklist to run metadata check after tag push.

`docs/product/2026-05-30-handoff.md`
- Record `REL-003` completion and verification.

Forbidden:

- `prepublishOnly` must not call `check:release:metadata`.
- `release`, `release:minor`, and `release:major` must not call `check:release:metadata`.
- `dist/` must not be manually edited.
- `plugins/codex-claude-delegate/server/*.js` must not be manually edited.

## 3. Positive Acceptance Cases

Scenario: Registry and tag match local version.

Input:

```json
package.json: { "name": "@scope/pkg", "version": "1.2.3" }
npm view JSON: { "version": "1.2.3", "dist-tags": { "latest": "1.2.3" } }
git tags: "v1.2.3\n"
```

Expected:

- `node scripts/check-release-metadata.mjs` exits 0.
- stdout contains `check:release:metadata ok (@scope/pkg v1.2.3)`.

Scenario: Local dry run skips external checks explicitly.

Input:

```text
CHECK_RELEASE_SKIP_NPM=1
CHECK_RELEASE_SKIP_GIT=1
```

Expected:

- Script exits 0 when local `package.json` has valid name/version.
- stdout mentions skipped checks.

## 4. Negative Acceptance Cases

Scenario: package metadata missing.

Input:

```json
package.json: { "name": "@scope/pkg" }
```

Expected:

- Exit code 1.
- stderr contains `package.json missing name or version`.

Scenario: Registry version mismatches.

Input:

```json
package.json version: "1.2.3"
npm view JSON: { "version": "1.2.2", "dist-tags": { "latest": "1.2.3" } }
```

Expected:

- Exit code 1.
- stderr contains `registry version 1.2.2 !== package.json 1.2.3`.

Scenario: Registry latest mismatches.

Input:

```json
package.json version: "1.2.3"
npm view JSON: { "version": "1.2.3", "dist-tags": { "latest": "1.2.2" } }
```

Expected:

- Exit code 1.
- stderr contains `registry dist-tags.latest 1.2.2 !== package.json 1.2.3`.

Scenario: Git tag missing at HEAD.

Input:

```text
package.json version: 1.2.3
git tag --points-at HEAD v1.2.3 returns empty stdout
```

Expected:

- Exit code 1.
- stderr contains `git tag v1.2.3 is not present at HEAD`.

Scenario: npm view output is invalid JSON.

Expected:

- Exit code 1.
- stderr contains `npm view returned invalid JSON`.

## 5. Forbidden Behaviors

- Do not call network-dependent registry checks from `prepublishOnly`.
- Do not silently pass when registry output is missing `version` or `dist-tags.latest`.
- Do not accept any tag name except exact `v<package.json version>` at HEAD.
- Do not read the user's global npm config in tests.
- Do not mutate files, tags, branches, or registry state.
