# MULTI-ROOT-002 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Strengthen context-root provenance guidance so query and review outputs cite additional repositories by alias and path, with tests focused on review output prompts.

**Architecture:** Keep runtime behavior unchanged. Extend the shared context-roots prompt section in `src/schema.ts`, because both `buildQueryPrompt` and `buildReviewPrompt` consume it. Add prompt-level tests proving review and query prompts require alias-qualified citations for context-root evidence.

**Tech Stack:** TypeScript prompt builders, Vitest unit tests, Markdown README.

---

## 1. Background And Goal

Current behavior:

- `context_roots` lets `claude_task`, `claude_query`, and `claude_review` include extra read-only repositories.
- The prompt lists aliases and paths, and says context roots are read-only.
- Review output guidance does not explicitly require findings to identify whether evidence came from the primary repo or a context root.

Target behavior:

- The context roots prompt must require source provenance for any claim based on context-root content.
- The required citation form must include the context root alias and a file path or command source.
- Review prompts must make this especially clear for findings/recommendations, so review output can distinguish primary repo issues from context-root evidence.

Out of scope:

- Do not change `context_roots` validation rules.
- Do not change Claude CLI permission allowlists.
- Do not change JSON output schemas.
- Do not add post-processing that validates Claude's natural-language citations.
- Do not hand-edit `dist/` or generated plugin server bundles.

## 3. Positive Acceptance Cases

Scenario: Shared context roots prompt lists provenance rules.

Input:

```ts
buildContextRootsPromptSection([{ alias: "lib", cwd: "/repos/lib" }])
```

Expected prompt contains:

- `## Context Roots`
- `**lib**`
- `/repos/lib`
- `Cite context-root evidence`
- `[lib]`
- `Do NOT modify any files in context roots`

Scenario: Review prompt with context roots requires alias/path citations.

Input:

```ts
buildReviewPrompt({
  cwd: "/repo",
  task: "review cross-repo integration",
  context_roots: [{ alias: "lib", cwd: "/repos/lib" }],
})
```

Expected prompt contains:

- Context roots section.
- A review-specific instruction that findings based on context roots must name the alias and path.
- Existing reviewer instructions remain present.

Scenario: Query prompt with context roots inherits provenance rules.

Input:

```ts
buildQueryPrompt({
  cwd: "/repo",
  task: "explain cross-repo integration",
  context_roots: [{ alias: "lib", cwd: "/repos/lib" }],
})
```

Expected prompt contains:

- Context roots section.
- Alias/path citation instruction.

## 4. Negative Acceptance Cases

Scenario: Prompt without context roots remains unchanged for provenance.

Input:

```ts
buildReviewPrompt({ cwd: "/repo", task: "review" })
buildQueryPrompt({ cwd: "/repo", task: "explain" })
```

Expected:

- No `## Context Roots` section.
- No context-root citation instruction.

Scenario: Provenance guidance must not imply context roots are writable.

Expected:

- Prompt still contains `Do NOT modify any files in context roots`.
- No instruction asks Claude to edit, patch, apply, or commit in a context root.

## 5. Forbidden Behaviors

- Do not make context-root citation optional for claims based on context-root files or command output.
- Do not require citations for claims based only on the primary repo.
- Do not blur context roots with primary cwd; wording must distinguish them.
- Do not change `REVIEW_SCHEMA` required fields.
- Do not weaken sensitive file deny guidance.
